Programming Defensively

Very often we write quick little programs to work on data that we know conforms to a specific format, or that will read files that we know will exist and that we have permissions to open and read from or write to. But, relying on those assumptions isn't good practice. The first suggestion is to check the return value of system calls like the open() function: 

    
    open(FILE, $file) || die "Can't open $file: $!"; # good
    open FILE, $file  or die "can't open $file: $!"; # good
    open(FILE, $file);  # not good

The open() function returns a false value if it fails, so in the first two examples above, if the call fails the die() function is called. We often don't want to continue with the program if we couldn't open the file, but you could do other things if that's more appropriate for your program. If you've used Perl for very long, read the docs, or spent any time reading the comp.lang.perl.misc newsgroup then you've probably seen this advice before. Even in short little programs or examples this a good thing to do (it should be mandatory). However, defensive programming is not limited to checking the return value of the open() call.
